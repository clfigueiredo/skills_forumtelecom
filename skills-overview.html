01 / ARQUITETURA

O que é uma skill, mecanicamente

Uma skill é um arquivo SKILL.md que vira contexto para o LLM. O agente lê o frontmatter, decide se a skill é relevante para o pedido do usuário, carrega o conteúdo e segue o playbook. Sem código compilado, sem black-box.

Anatomia de um SKILL.md

O frontmatter YAML define name e description — densos em palavras-gatilho. Quanto melhor a description, melhor o agente decide quando ativar.

O metadata.openclaw declara binários e env vars exigidos. O OpenClaw usa para gating; o Claude SDK ignora silenciosamente.

O corpo em markdown é o playbook — instruções claras, comandos críticos, regras de segurança, templates. O agente lê isso e executa.

---
name: mikrotik-ops
description: Senior MikroTik RouterOS engineer.
  Use when user asks to diagnose, configure, audit
  MikroTik devices. Triggers: BGP, hotspot, /interface,
  CCR, CRS, RouterOS 6 vs 7, queue tree...
metadata: { "openclaw": { "emoji": "🛜",
  "requires": { "anyBins": ["ssh"],
  "env": ["MIKROTIK_HOST"] } }

02 / WORKFLOW

O ciclo de 5 etapas obrigatórias

Toda skill obriga o agente a seguir o mesmo ritual antes de mexer em produção. Não é decoração — é o que separa um agente que ajuda de um agente que derruba a rede do cliente.

01
Identify
Versão do firmware, hostname, contexto do equipamento.
02
Snapshot
Backup ou export antes de qualquer mudança.
03
Apply
Aplica com safe-mode, commit-confirmed ou scheduler de rollback.
04
Validate
Confere que a mudança fez efeito esperado.
05
Report
Relatório markdown estruturado do que foi feito.
SAFETY · CONFIRMATION PATTERN
Ops destrutivas exigem confirmação literal

Para qualquer operação que pode quebrar produção (apagar regra, derrubar peer BGP, deletar VM, dar reload, factory reset), o agente recusa até receber do usuário a frase exata CONFIRMO <comando> — paráfrases como "ok pode mandar" ou "manda ver" são rejeitadas. Isso é instrução fixa em todas as 8 skills.

SKILL #01 · NETWORK
mikrotik-ops
"Engenheiro sênior de RouterOS, 24/7."
Opera RouterOS via SSH ou REST API. Conhece a fundo as diferenças entre RouterOS 6 e 7 (BGP, IPv6, container, ROSE). Aciona modo seguro para mudanças remotas usando /system scheduler com auto-rollback. Domina os 3 papéis típicos do MikroTik em provedor: borda BGP, BRAS PPPoE e router de cliente.
Casos típicos no provedor
  • BRAS PPPoE com queue tree Mangle marca conexão por interface dinâmica → queue tree HTB aplica plano de banda. Nunca queue simples em concentrador grande — vira SMP-bound.
  • CGNAT NAT44 com pool Provedor sem IPv4 público suficiente. Pool de saída via action=src-nat to-address-list + log mínimo de 12 meses (compliance).
  • Borda BGP múltiplos upstreams RouterOS 7 mudou tudo: /routing/bgp/connection, templates, filter rules separados. Diagnóstico de queda começa em session + log topics=bgp.
  • Failover ECMP recursive Em v7, target-scope recursivo + check-gateway. Em v6, PCC mangle. Migrar pra v7 simplifica drasticamente.
Prompts típicos no WhatsApp
BGP do upstream Vivo caiu Cliente X não navega, dá uma olhada Sobe a banda do cliente Y pra 500M Lista quem tá logado no concentrador Bloqueia a porta 25 saída do IP 10.0.0.5
Equipamentos
CCR2004 CCR2216 CRS hAP RouterOS 6 RouterOS 7 REST API CAPsMAN

$MIKROTIK_HOST · $MIKROTIK_USER · $MIKROTIK_PORT · $MIKROTIK_API_PASS

SKILL #02 · NETWORK
cisco-ops
"Detecta o OS, fala a sintaxe certa."
Roteadores e switches Cisco em qualquer sabor. Detecta automaticamente IOS / IOS-XE / IOS-XR / NX-OS via show version e ajusta a sintaxe. Cada plataforma tem seu mecanismo de safety: IOS-XR usa commit confirmed 5, NX-OS usa checkpoint, IOS clássico precisa de reload in 10 reservation.
Casos típicos no provedor
  • Borda BGP full-table múltiplos upstreams ASR9K recebendo ~950k IPv4 + ~200k IPv6. maximum-prefix 1500000 90 restart 5 obrigatório. Diagnóstico padrão: bgp summary + received-routes count + platform memory.
  • CGNAT em IOS-XR (CGv6) Line-card dedicada (NCS5500 + ServiceInfra). Pool, timers TCP 120s, export Netflow v9 para retenção legal de 12 meses.
  • MPLS L3VPN para clientes corporativos VRF + RD + RT + BGP vpnv4. Diagnóstico de cliente: bgp vpnv4 vrf, mpls forwarding-table vrf, ip cef vrf prefix.
  • Anti-DDoS via blackhole BGP Community do upstream injeta nullroute em /32 alvo. Padrão temporário — manter permanente vira buraco de alcance.
Prompts típicos no WhatsApp
Peer X dropou, vê o motivo Latência alta no link Y, investiga Anuncia nullroute pro IP Z (atacado) Backup da config pro bootflash Cliente VPN X sem conectar
Plataformas
ASR9K ASR1K ISR4K Catalyst 9K Nexus 9K IOS-XR NX-OS
Env vars
$CISCO_HOST · $CISCO_USER · $CISCO_PORT

SKILL #03 · NETWORK
huawei-ne-ops
"VRP V5 ou V8 — o agente sabe a diferença."
Roteadores de borda Huawei NetEngine. VRP V8 (NE40E moderno, NE8000) usa commit — V5 (NE40E classic, ME60) aplica direto. O agente detecta e ajusta. Conhece display em vez de show, undo em vez de no, e o ritual completo de BNG (aaa scheme + domain + virtual-template + IP pool).
Casos típicos no provedor
  • BNG PPPoE de larga escala AAA scheme com RADIUS, domain por ISP/marca, virtual-template, IP pool com network /16 (CGNAT). Diagnóstico de cliente: display access-user username + radius-server statistics.
  • CGNAT em placa CGN dedicada Service-location apontando para slot com card CGN. NAT instance + address-group + outbound rule. Export syslog/Netflow obrigatório.
  • Peering BGP com BFD BFD acelera detecção de falha (300ms vs 90s do hold-timer). Comum em peering de upstream crítico.
  • L3VPN para empresas VPN-instance + RD + vpn-target + binding em subinterface dot1q. Sintaxe diferente do Cisco mas conceito idêntico.
Prompts típicos no WhatsApp
Cliente PPPoE não autentica Quantos assinantes online agora? BGP do upstream Y caiu Cliente Z conecta mas sem IP Backup da config pro FTP
Plataformas
NE40E NE8000 NE20E ME60 VRP V5 VRP V8 BNG
Env vars
$HUAWEI_HOST · $HUAWEI_USER · $HUAWEI_PORT

SKILL #04 · FTTx
olt-fiberhome-ops
"CLI por diretórios — o agente navega como filesystem."
OLTs FiberHome têm CLI peculiar — você navega entre cd device, cd vlan, cd gponline, e o prompt muda. Comando válido em um diretório não funciona em outro. O agente sempre confere o prompt antes de executar. Domina autoprovision via show discovery + whitelist.
Casos típicos no provedor
  • Provisionamento massivo de ONUs ONU física conecta → aparece em show discovery → whitelist com SN → service-port com VLAN PPPoE. Integração TL1 na porta 3334 para sistemas de gestão.
  • Sinal ótico (80% dos chamados) OLT RX e ONU RX ideais entre -8 e -27 dBm. show optic_module + interpretação dB. Distância confronta com cadastro (suspeita de fraude se diverge).
  • 3rd-party ONUs (BDCOM, Raisecom, etc.) Compatibilidade depende do firmware: RP0700 restritivo, RP1000+ amplo. Diagnóstico: SN aparece em discovery? Se sim → vendor não suportado pela placa.
  • Suspensão por inadimplência Não delete a ONU — perde histórico. Move pra VLAN de "suspensos" via service modify. Reativação reverte VLAN.
Prompts típicos no WhatsApp

Plataformas

SKILL #05 · FTTx
olt-huawei-ops
"VRP-style com modo enable, config, interface."
OLTs Huawei (MA5800, MA5680T, EA5800) usam CLI VRP-derivado: user view, enable view, config view, e contextos de placa GPON. Provisionamento de ONT segue o padrão F/S/P (frame/slot/port). O agente conhece line-profile + srv-profile + service-port + traffic-table como combinação que define o serviço.
Casos típicos no provedor
  • Autoprovision via auto-find ONT aparece em display ont autofind → vincula a profiles → cria service-port com tag-transform. Provedores grandes têm dezenas de profiles para diferentes planos.
  • Tag-transform: VLAN cliente vs core transparent / translate / add (Q-in-Q) / default. Q-in-Q comum quando OLT só transporta L2 e BNG termina PPPoE.
  • Localização de cliente por SN display ont info by-sn retorna F/S/P + ID. Depois display ont event dá últimos eventos (LOS, dying-gasp, etc.).
  • Diagnóstico de sinal ótico RX/TX em display ont optical-info. Run-state online/offline/failing. Distance em metros (validar com cadastro).
Prompts típicos no WhatsApp

SKILL #06 · INFRA
proxmox-ops
"qm, pct, pvecm, pvesh, pvesm — domínio completo."
Cluster Proxmox VE: VMs (KVM via qm), CTs (LXC via pct), cluster (pvecm), storage (pvesm), API (pvesh). Snapshot antes de qualquer mudança — sempre. Conhece arquitetura típica de provedor: 3 nodes, redes redundantes para corosync, PBS dedicado para retention longa.
Casos típicos no provedor
  • Hospedagem de stack interna Sistema de gestão (IXC, Hubsoft, MK-Auth), portal cliente, RADIUS, looking glass, Zabbix — tudo em VMs/CTs num cluster 3-node com Ceph ou ZFS replicado.
  • Cluster 3-node com HA real --link0 + --link1 (redes redundantes para corosync) é OBRIGATÓRIO. Sem isso, qualquer hiccup de rede causa fence indevido.
  • Replicação ZFS (alternativa Ceph) Pra clusters 2-3 nodes sem stack Ceph completa. pvesr a cada 15min, failover manual em segundos.
  • Backup com PBS + dedup PBS faz deduplicação real. Backup full diário ocupa pouco espaço incremental. Retention 30+ dias vira simples.
Prompts típicos no WhatsApp
Como tá o cluster? VM do IXC tá lenta, investiga Restaura backup de ontem da VM 105 Snapshot antes do upgrade Quanto storage livre tem?
Cobertura
VM (KVM) CT (LXC) Cluster ZFS Ceph PBS HA
Env vars
$PVE_HOST · $PVE_TOKEN (API token)

SKILL #07 · INFRA
docker-ops
"Containers, Compose, Swarm, Traefik."
Operações de container em qualquer contexto: socket local, SSH remoto, Coolify, Portainer, Komodo. Sempre prefere docker compose declarativo a docker run ad-hoc. Healthchecks obrigatórios — container sem healthcheck é container que parece up mas tá quebrado.
Casos típicos no provedor
  • Stack de plataforma interna Portal cliente, looking glass, monitoramento, automação, agentes de IA — tudo em containers com Traefik + Let's Encrypt + Cloudflare Tunnel.
  • Pattern Traefik com HTTPS automático Labels declarativas: rule, certresolver, healthcheck path/interval. Cert gerado automaticamente, renovado em background.
  • Diagnóstico "container ON mas não responde" docker ps + logs --tail + exec ps aux + stats + network inspect + port. Causas comuns: healthcheck falhando, OOM kill, volume com permissão errada.
  • Backup de volumes nomeados tarball via container alpine + cp out + restic/borg para dedup ofsite. Combinar com snapshot Proxmox da VM toda.
Prompts típicos no WhatsApp
Container X caiu, verifica Deploy da nova versão do portal Backup do banco postgres Reinicia stack X Quem tá comendo CPU agora?
Contextos
Single-host Swarm Coolify Portainer Traefik Compose

SKILL #08 · OBSERVABILITY
zabbix-ops
"API JSON-RPC com rate limit respeitado."
Monitoramento Zabbix 6.x e 7.x via API JSON-RPC. Especialidade: criação de templates em massa respeitando os limites práticos da API — PHP-FPM workers, max_input_vars, locks de Postgres. Domina batches sequenciais, retry exponencial, e quando usar configuration.import vs item.create em loop.
Casos típicos no provedor
  • Criar template com 200 items + 50 triggers Items em batches de 50 (até 50/call), triggers em batches de 30, throttle de 200ms entre chamadas, retry exponencial em -32500 e -32603. Sem isso → frontend trava.
  • Vincular template a 200 hosts Use template.massadd (transacional, até 500 hostids/call) ao invés de loop de host.update. 1 chamada vs 200 chamadas.
  • Discovery automático de equipamentos Range de IP gerência → drule.create → action de auto-add com template baseado em sysObjectID SNMP. Network novo entra sozinho no monitoramento.
  • Webhook de alerta → WhatsApp Mediatype JS chamando Evolution API. Severity ≥ High dispara mensagem no grupo NOC. Acknowledge fecha o problema.
  • Replicação de templates entre instâncias configuration.export YAML na origem + configuration.import no destino. Ideal para replicar customizações entre Zabbix de homolog/prod.
Prompts típicos no WhatsApp
Quem tá com problema agora? Cria template MikroTik com 50 itens Importa esse XML do template Huawei Vincula template X a esses 200 hosts Silencia alerta da OLT durante manutenção Tráfego do uplink no último dia
Limites respeitados
item.create ≤50 trigger.create ≤30 host.create ≤20 throttle 200ms paralelismo 3-5

04 / INTEGRAÇÃO

Como tudo isso roda no NetAgent

O NetAgent recebe mensagem do WhatsApp via Evolution API, decide qual agente especializado deve responder (Network ou Infra), e o agente carrega as skills relevantes do diretório montado.

01
User → WhatsApp
Usuário manda mensagem tipo "o BGP do MikroTik de borda caiu, dá uma olhada". Evolution API entrega o webhook ao FastAPI do NetAgent.
02
Dispatcher → Agent
O Dispatcher classifica a intenção (network) e roteia para o NetworkAgent. O agente é instanciado com o diretório de skills montado em /app/skills.
03
Agent → Skill
O Claude Agent SDK lê o frontmatter de cada SKILL.md, decide que mikrotik-ops é a skill certa, carrega o playbook e executa o workflow de 5 etapas via SSH/MCP.

Código de exemplo · NetworkAgent 

# netagent/agents/network_agent.py
from claude_agent_sdk import query, ClaudeAgentOptions

class NetworkAgent:
    def __init__(self):
        self.options = ClaudeAgentOptions(
            system_prompt="Você é o NetworkAgent da Fórum Telecom.",
            setting_sources=["project"],          # carrega skills do dir
            cwd="/app/skills",                      # monta forum-telecom-skills aqui
            allowed_tools=["Bash", "Read", "WebSearch"],
            permission_mode="default",             # exige confirmação humana
        )

    async def handle(self, user_message: str):
        async for msg in query(prompt=user_message, options=self.options):
            yield msg

PRO TIP · FONTE ÚNICA DE VERDADE
Mantenha as skills num diretório versionado

Tanto o NetAgent (Claude Agent SDK) quanto o OpenClaw consomem os mesmos arquivos. Mantenha um único diretório como fonte da verdade e os dois runtimes apontam pra ele. Toda alteração de comando ou novo template fica num lugar só — sem divergência entre ambientes.
